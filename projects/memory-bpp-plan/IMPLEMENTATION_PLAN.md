# 实施计划（分阶段执行 + 逐步验收）

## 1. 执行原则

1. 每步先小范围实现，再验证通过后进入下一步。
2. 每步均可回滚，不影响主流程可用性。
3. 每步都必须产出可审计日志。

## 2. 阶段总览

1. Phase A：基础能力（分类 + 检索链路）
2. Phase B：写回治理（质量门控 + 去重）
3. Phase C：强化淘汰（usage + lifecycle）
4. Phase D：评测上线（A/B + 门禁）

## 3. 详细步骤

## Step 0: 基线冻结

### 输入

1. 当前 `memory-md-index` 插件代码。
2. 当前配置与运行脚本。

### 要求

1. 记录当前默认行为与关键指标（成功率、token、P95）。
2. 固化基线配置，生成 `baseline` profile。

### 产出

1. 基线报告（至少 20 条样本）。
2. 基线配置文件快照。

### 验收标准

1. 可重复执行，重复两次偏差在可接受范围（成功率偏差 <= 5%）。

---

## Step 1: 数据模型升级（frontmatter + 目录分类）

### 输入

1. 现有 `memory/short|mid|long` 文件。

### 要求

1. 增加 `mid/<domain>/` 分类目录。
2. 写回条目统一 frontmatter 最小字段。
3. 保持旧条目兼容读取（缺字段时默认值补齐）。

### 产出

1. 目录迁移脚本或兼容读取逻辑。
2. 更新后的 schema 文档和样例文件。

### 验收标准

1. 新旧条目都可被 `memory_search` 正常检索。
2. 写回不报错，字段完整率 >= 95%。

---

## Step 2: 路由提示与预过滤（before_route）

### 输入

1. 用户 query。
2. 可选上下文（session/domain hints）。

### 要求

1. 输出 `domain/allow_tags/deny_tags/priority_paths`。
2. 检索前执行候选预过滤，保留兜底召回窗口（例如额外 1~2 条跨域）。

### 产出

1. `before_route` 实现。
2. 路由日志（含命中域和过滤规模）。

### 验收标准

1. 歧义样本误召回率较 Step 0 下降 >= 20%。
2. 漏召回率不上升超过 5%。

---

## Step 3: 检索重排与注入预算（before_prompt_build）

### 输入

1. 预过滤候选。
2. 检索后端结果（rg/BM25/vector）。

### 要求

1. 实现统一重排公式并记录每项得分。
2. 注入 `topK` 与 `maxChars` 双预算限制。
3. 注入块含来源引用与截断标记。

### 产出

1. 重排模块与 debug 日志。
2. Prompt 注入规范（固定模板）。

### 验收标准

1. 有效引用率较 Step 2 提升 >= 10%。
2. token 增幅控制在 <= 20%。

---

## Step 4: 写回质量门控与去重（agent_end）

### 输入

1. `result.final`、可选 trace、注入记忆列表。

### 要求

1. 抽取结构化条目（Fact/Decision/TODO/RuleCandidate）。
2. 写回前进行噪声过滤与去重。
3. `short` 覆盖更新，`mid` 按域追加，`long` 只写 proposal。

### 产出

1. 写回策略模块。
2. 去重报告（每日）。

### 验收标准

1. 重复率较 Step 0 下降 >= 30%。
2. 无明显无效文本污染（抽样 100 条污染率 < 5%）。

---

## Step 5: usage 事件与生命周期评分

### 输入

1. retrieve/inject/outcome 事件。

### 要求

1. 建立 `usage.jsonl` 事件日志（append-only）。
2. 每日计算 `lifecycle_score` 并输出决策（keep/promote/archive）。
3. 强化必须绑定结果信号（success/fail/correction）。

### 产出

1. lifecycle scorer。
2. lifecycle 决策日志与解释字段。

### 验收标准

1. 低价值记忆占比逐周下降。
2. 错误强化率（被纠正后仍升权）< 2%。

---

## Step 6: 定时维护任务（daily/weekly）

### 输入

1. memory 数据目录。
2. usage/lifecycle 日志。

### 要求

1. `daily`：去重、归档、增量索引。
2. `weekly`：规则提纯与冲突报告。
3. 故障自动降级，不阻塞主流程。

### 产出

1. maintenance 命令与调度配置。
2. 每日/每周报告文件。

### 验收标准

1. 任务成功率 >= 99%。
2. 单次 daily 在目标资源下耗时可控（例如 < 5 分钟）。

---

## Step 7: A/B 评测与门禁决策

### 输入

1. `EVALUATION_PLAN.md` 中定义的数据集和指标。

### 要求

1. 完整执行 G0/G1/G2/G3。
2. 输出组间对比报告与置信区间。
3. 按门禁规则给出 Go/No-Go。

### 产出

1. 最终评测报告。
2. 上线建议与回滚预案。

### 验收标准

1. 达成评测门禁条件才启用默认配置。
2. 未达标时必须落回上一稳定配置。

## 4. 依赖与资源

1. 必需：Node 22+、ripgrep、现有 OpenClaw 插件运行环境。
2. 推荐：BM25 本地索引或可替换 vector 命令。
3. 可选：仪表盘脚本（生成日报/周报）。

## 5. 回滚策略

1. 全局回滚：禁用 memory 插件。
2. 局部回滚：关闭 lifecycle，仅保留检索注入。
3. 后端回滚：vector -> bm25 -> rg 自动降级。

## 6. 里程碑建议

1. M1（3 天）：完成 Step 1~3，具备稳定检索注入。
2. M2（7 天）：完成 Step 4~6，具备治理闭环。
3. M3（14 天）：完成 Step 7，通过门禁后上线。
